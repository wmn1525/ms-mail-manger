# 开发指南

本文件为 Codex 在本仓库工作时的项目级规则。若与用户当前明确指令冲突，优先遵循用户当前指令；若与更高优先级系统规则冲突，优先遵循更高优先级规则。

## 语言规则

- 默认使用简体中文与用户沟通，包括解释、计划、任务、总结和错误说明。
- 文档内容默认使用简体中文。
- 代码、命令、文件路径、配置键、API 名称、错误信息和日志内容保持原文，不强行翻译。
- 函数名、方法名、变量名、类名等标识符可保留英文。
- 代码注释使用中文，并说明意图、约束或关键原因，避免只重复代码表面语义。
- 生成文档或注释时，优先遵循目标文件已有语言风格；没有既有风格时使用简体中文。
- 涉及英文技术术语时，可保留英文原词，并在必要时补充中文解释。

## 工程硬性规则

- 不添加 fallback、双写、临时适配、过渡 API、旧新结构转换壳等兼容层。
- 禁止使用 `any`、`as any`、宽泛 `unknown`、双重断言，禁止用 `Record<string, unknown>` 代替业务 DTO。
- 文件级注释、类、类属性、方法、函数、类型字段和复杂逻辑需要有中文注释说明；注释必须服务于理解意图、约束或关键原因。
- 单个代码文件总行数不得超过 400 行，统计范围包括代码、空行、注释、类型定义、常量、样式和模板内容。
- 工具类和配置文件建议控制在 200 行以内；超过上限时按职责拆分。
- 新增或修改后的文件一旦超过 400 行，必须立即拆分，不得以“后续优化”“临时过渡”“当前先这样”为理由保留超限文件。
- 拆分必须按职责边界进行，优先拆分为类型定义、常量配置、纯工具函数、业务编排、UI 展示、样式等独立文件。
- 不得通过压缩代码、删除必要注释、合并无关职责、堆叠复杂函数等方式规避行数限制。
- 自动生成文件、依赖目录、构建产物、锁文件不适用行数限制，但不得手写维护超限生成文件。




## 开发行为准则

### 编码前先澄清

- 不做隐含假设；如果需求存在多种解释，先说明并询问。
- 如果有更简单方案，主动指出权衡。
- 如果信息不足，停止实现并说明缺口。


- Prefer standard library first.
- Prefer mature third-party libraries over custom implementations.
- Never reinvent the wheel.
- Keep code minimal.
- Minimize token usage.
- Return only the necessary code.
- Avoid explanations unless requested.
- Modify existing code instead of rewriting files.
- Prefer configuration over implementation.
- Prefer composition over abstraction.

### 简单优先

- 不实现未被要求的功能。
- 不为单次使用创建抽象。
- 不添加未被要求的配置化或扩展点。
- 不为不可能发生的内部场景添加防御式分支。
- 如果实现明显可以大幅简化，应先简化再提交。

### 外科手术式修改

- 只修改完成任务必须修改的代码。
- 不顺手重构无关代码，不清理既有无关死代码。
- 匹配现有风格，即使有其他个人偏好。
- 由本次修改造成的未使用导入、变量、函数必须清理。
- 如果发现无关死代码或风险，说明即可，不要擅自删除。

### 目标驱动并验证

- 将任务转为可验证目标。
- 修复 bug 时优先写出或定位能复现问题的检查方式。
- 完成实现后运行与变更范围匹配的测试、类型检查、lint 或专项校验。
- 多步骤任务应给出简短计划，并为每一步明确验证方式。
